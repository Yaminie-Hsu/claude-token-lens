# 设计决策记录

## 核心定位

**补充而非替代**。Claude Code 官方已有 `/compact`、`/clear`、自动压缩、Prompt 缓存等机制。
本工具处理更早的环节——Prompt 提交之前，和工具执行之前。与官方机制形成完整的节省链路。

---

## 决策 1：Hook 接入方式

**选择**：使用 Claude Code 官方 `UserPromptSubmit` 和 `PreToolUse` hook。

**排除方案**：
- 独立 CLI 工具（需要用户手动调用，摩擦大）
- 代理服务（复杂，有安全顾虑）

**理由**：原生集成是被官方注意到的最直接方式；零摩擦是用户采用的关键。

---

## 决策 2：编程场景三种压缩策略

**选择**：stack-trace 压缩 → code-block 清理 → whitespace 规范化，顺序执行。

**核心洞察**：编程时 token 浪费的三大来源：
1. 长堆栈报错（粘贴整个 traceback，只需头尾帧）
2. 注释密集 / 空行过多的代码块
3. 正文里的大量空行

**参数设计**：
- `stack_head_lines=3, stack_tail_lines=5`：头部少（定位入口），尾部多（出错点）
- `strip_code_comments` 默认关闭：注释有时有意义，用户自行选择
- `report_threshold_tokens=20`：低于此值静默，避免噪音

---

## 决策 3：文档处理中的页码策略（关键）

**背景**：最初设计是"去掉页码"作为清洗步骤。

**用户纠正**（来自律师使用场景）：
> 页码是导航锚点，是连接 AI 输出和原始 PDF 的唯一线索。
> 解析平台返回 Markdown 时页面边界已丢失。
> AI 返回引用片段后，律师必须能翻到原文对应页。

**最终决策**：
- **保留并强化**页码，而非去除
- 提取时在每页边界插入结构化标记：`─── [第 N 页] ────────`
- 去除的是**重复的页眉页脚文字**（如"保密文件 XX公司 2024"），通过跨页重复检测识别
- `--keep-headers` 参数供用户禁用这一步

**影响**：这个决策是文档模块最重要的设计原则，所有后续模块（cleaner、chunker、timeline）都围绕"页码必须可追溯"展开。

---

## 决策 4：时间线功能

**背景**：律师工作中有一个核心需求——从多份文档里梳理时间线（谁在什么时间做了什么）。这是最耗时的手工工作之一。

**设计原则**：
- 每个时间节点必须带**来源文件名 + 页码**，否则对律师没有实用价值
- 支持中文日期（`2024年3月15日`）、ISO 格式（`2024-03-15`）、英文格式（`March 15, 2024`）
- 跨多文档合并，按时间排序
- 输出为 Markdown 表格，可直接贴入工作记录

**技术坑**：Python 3 的 `re` 模块中，`\b` 把中文字符也视为 `\w`，导致 ISO 日期在中文语境里匹配失败。
解决方案：用 `(?<!\d)` 和 `(?!\d)` 替代 `\b`。

---

## 决策 5：Token 计数方案

**选择**：`tiktoken` (cl100k_base)，不可用时退化到 `len(text) // 4`。

**理由**：Claude 的 tokenizer 与 GPT-4 接近；tiktoken 是开源标准库；退化方案保证无依赖时也能运行。

---

## 已知局限

- 注释检测是语言无关的正则（`//` 和 `#`），可能误删某些非注释内容
- 堆栈帧检测覆盖主流语言，Ruby/PHP/Swift 等尚未验证
- 文档关键字段提取（甲方、合同金额等）为规则匹配，仅适用于中文合同格式
- `setup` 命令的路径处理尚未在 Windows 上测试
